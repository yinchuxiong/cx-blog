## 一、Letta（原 MemGPT）

### 1.1 概述

Letta 的前身是 UC Berkeley 2023 年的学术项目 MemGPT，后改名 Letta 商业化。它是一个**自带记忆管理层的 LLM 运行时**，三段记忆分层是其核心设计。

### 1.2 三段记忆架构

| 记忆层 | 类比 | 存储后端 | 用途 |
|--------|------|----------|------|
| **Core Memory**（核心记忆） | RAM / 寄存器 | 关系型数据库（PostgreSQL / SQLite） | 跨会话持久化的关键事实摘要，直接注入 System Prompt |
| **Recall Memory**（召回记忆） | 事件日志 | 关系型数据库（PostgreSQL / SQLite） | 完整对话历史，按时间排序 |
| **Archival Memory**（归档记忆） | 硬盘 / 冷存储 | 向量数据库（Qdrant / ChromaDB 等） | 长期知识库，语义相似度检索 |

### 1.3 Core Memory 详解

- 不是一个会话的上下文，而是**所有历史会话中被提炼出的关键事实摘要**
- 每次新会话启动时都会从数据库加载，**跨会话持久化**
- 结构上分两块：
  - **Persona Block**：Agent 的身份、性格、行为准则
  - **Human Block**：Agent 对用户的认知（偏好、背景、当前项目等）
- **自编辑机制（Self-Editing）**：Agent 每轮对话后做内部反思（introspection），自主判断是否需要更新 Core Memory，调用 `core_memory_replace` / `core_memory_append` 工具修改。不是基于关键词规则匹配，而是靠 LLM 的语义理解判断

### 1.4 Recall Memory 详解

- 存放**原始对话记录**，不做语义分析、不做 embedding、不做提炼
- 按时间顺序追加，不可修改、不可删除
- 不像 Core Memory 那样主动注入上下文，只在需要回顾历史时才查询
- 和 Archival Memory 的区别：Recall 按时间查（"上周二聊了什么"），Archival 按语义查（"之前讨论的缓存方案是怎样的"）

### 1.5 Archival Memory 详解

- 本质上是 RAG 知识库：embedding + 向量检索
- 三种写入来源：
  1. 用户手动存入（上传 PDF、粘贴文档等）
  2. Agent 主动存档（判断某些结论日后可能用得上）
  3. Core Memory 溢出转移（上下文满了，旧的高价值信息转到此处）
- 支持按 source / tag / namespace 分区隔离
- **已知缺陷**：Agent 自动存档可能不准确（摘要偏差），有可能造成知识污染。严肃场景下建议关掉 Agent 自动写入权限，仅保留用户手动存 + Agent 只读检索

### 1.6 记忆更新触发机制

三种触发条件：

1. **上下文窗口快满**：对话历史 + Core Memory + 检索结果接近 token 上限时，触发记忆整理
2. **Agent 主动判断有重要信息**：每轮对话后反思是否需要持久化新事实
3. **用户显式触发**：说"记住这个"或"存起来"

### 1.7 上下文压缩机制

压缩的**不是 Core Memory**（Core Memory 本身已经很精炼），而是当前会话的原始对话记录：

- 把上下文中最旧的那批消息取出
- 让 LLM 压缩成简短摘要（如 10 轮对话压成 2-3 句话）
- 用摘要替换原始消息，释放上下文空间
- 原始消息保留在 Recall Memory 数据库中，只是不再占上下文

### 1.8 Letta 的核心价值

用了 Letta，等于**全套上下文管理都帮你做完了**。不需要自己处理截断策略、记忆维护、持久化、检索等。完全自托管部署，数据在自己手里，适合对数据隐私有刚性需求的场景（金融、医疗、政企）。

## 二、Mem0

### 2.1 定位

**记忆即服务**（Memory-as-a-Service）的云 API 产品。定位是快速接入，不用管底层部署。

### 2.2 架构特点

- **扁平架构**，没有 Letta 那样的三段分层
- 所有记忆平等对待，都在一个向量库
- 流程：对话文本 → LLM 提取事实 → embedding → 存向量库 → 检索时语义匹配召回
- 有去重合并机制（新事实与旧记忆意思差不多时合并，不新增）
- 按 `user_id` 隔离，不同用户记忆互不干扰

### 2.3 核心取舍

| Letta | Mem0 |
|-------|------|
| 自托管，数据在自己手里 | 云服务，数据在 Mem0 的服务器 |
| 三段分层，适合长期运行 | 扁平向量库，记忆量大时检索质量下降 |
| 运维成本高 | 接入成本低，几小时上线 |

### 2.4 适用场景

- 记忆量可控的轻量场景（几周到几个月的对话）
- 产品前期快速验证概念
- 不适合严肃的长期记忆场景，记忆量大后扁平架构的短板会暴露

## 三、Zep / Graphiti

### 3.1 定位

**时序知识图谱记忆**，核心差异化是双向时间建模。

### 3.2 双向时间建模

每条记忆标注 `valid_at`（何时生效）和 `invalid_at`（何时失效）。能回答"Agent 在上周二对这个事的认知是什么"这类回溯性问题。比如：
- 用户 3 月说"我在北京工作"→ valid_at = 3 月
- 用户 6 月说"我搬到上海了"→ 不是覆盖，而是把北京那条标上 invalid_at = 6 月
- 检索时知道现在在上海，也能回溯出 3-6 月期间在北京

### 3.3 优势

- 检索精度高：LongMemEval 基准上 71.2%，远超 Mem0 的 49%
- 检索延迟低：P95 约 300ms，检索时不调 LLM
- SOC 2 + HIPAA 合规
- 事实更新和冲突处理强

### 3.4 劣势

- **不支持自部署**：只提供托管云服务；开源版 Graphiti 需要自己搭 Neo4j，运维门槛高；旧开源社区版已废弃停止维护
- 没有 procedural memory（Agent 不能自己修改系统提示词）
- 计价按信用点数，不够透明
- 数据在云端，安全模型和 Mem0 本质上是同一类问题

## 四、Cognee

### 4.1 定位

**深度知识图谱引擎**，不是上下文管理框架。只做知识提取和结构化检索。

### 4.2 ECL 处理流水线

| 阶段 | 说明 |
|------|------|
| **E - Extract**（提取） | 从 38+ 种数据源摄入数据（对话文本、PDF、代码仓库等） |
| **C - Cognify**（认知化） | 调 LLM 自动提取实体和关系，构建语义知识图谱；同时做 embedding 存向量库 |
| **L - Load**（加载） | 数据分别存入图数据库（实体关系）+ 向量数据库（embedding）+ 关系型数据库（结构化元数据） |

### 4.3 存储两个层次

- **知识图谱层**：实体 + 关系（如"张三 → 工作于 → 字节跳动"）
- **原始内容层**：原始文本切片 embedding 存在向量库，用于原文片段召回

### 4.4 检索方式

14 种搜索模式，可混合使用：图遍历找关系 + 向量匹配找语义 + SQL 做精确过滤。擅长多跳推理和跨文档关联查询。

### 4.5 不做的事情

- 不做当前会话的对话列表维护
- 不做上下文窗口管理
- 不做对话压缩和截断
- 不做"昨天聊到哪了"的连贯性衔接

原始内容是服务于语义检索的素材切片，不是完整的对话审计日志。

### 4.6 实际落地方案

Cognee 不直接接在每条用户消息上。正规做法是在 Cognee 外面包一层会话管理层：

```
用户 → App（会话管理，存 Redis/Postgres）
           ↓
      对话结束后触发 Cognify（callback / cron）
           ↓
      Cognee 做 ECL → 存图谱 + 向量
           ↓
      检索时：先走自有向量库快速匹配 → 不够再走 Cognee 图谱深度关联
```

### 4.7 成本考量

Cognify 的提取 + 图谱构建比普通 embedding 贵得多，因为要调 LLM 做实体识别和关系抽取。大规模对话场景下（如万级用户）成本是严肃问题。实际做法是对话结束后批量异步处理，低峰期再合图去重。

### 4.8 适用场景

- 已有会话管理方案，只需要补强长期知识层
- 需要多跳推理、实体关系挖掘的场景
- 适合自部署、对数据安全有要求的企业场景（原生支持租户隔离 + 审计）

## 五、LangMem

### 5.1 定位

LangChain/LangGraph 的**原生记忆模块**，嵌在 LangGraph 状态图中运行，不是独立服务。不额外开服务、不额外连数据库（用 LangGraph 自带的 Store）。没用 LangGraph 的项目无法单独使用。

### 5.2 嵌入方式

LangMem 的函数直接作为状态图中的节点插入：

```
用户消息 → [LLM 生成回复] → [LangMem 提取记忆] → [返回给用户]
```

所有操作在同一个进程内完成，不走外部 API。

### 5.3 三种记忆类型

| 类型 | 存储方式 | 说明 |
|------|----------|------|
| **语义记忆** | 事实提取 + embedding + KV Store | 本质就是 RAG，和 Mem0 类似 |
| **情景记忆** | 对话消息原样存 Store | 按 user_id + 时间戳组织，和 Letta 的 Recall Memory 一致 |
| **程序记忆** | 自动修改 Agent 的系统提示词 | **LangMem 独有的能力** |

### 5.4 程序记忆（Procedural Memory）

LangMem 独有的机制——Agent 根据用户反馈**自动修改自己的系统提示词**，不需要人手动改：

- 用户说"你总是解释太啰嗦" → LangMem 提取反馈 → 自动追加规则"回答要简洁，不超过三句话"
- 下一轮对话新规则即生效

### 5.5 存储触发

提取是**异步后台运行**，不阻塞对话。对话结束后 LangMem 自动在后台调 LLM 提取新事实并写入 Store。不需要像 Cognee 那样自己写触发逻辑。

### 5.6 核心局限

- **没有图信息关系**，纯扁平 KV + 向量，无实体关联、无图谱遍历、无多跳查询
- 和 LangChain/LangGraph 强绑定，无法独立使用
- 无托管服务，需要自己运维基础设施
- 没有时序建模能力

## 六、横向对比总结

| 维度 | Letta | Mem0 | Zep | Cognee | LangMem |
|------|-------|------|-----|--------|---------|
| **定位** | 全包 Agent 记忆运行时 | 轻量记忆 API 服务 | 时序知识图谱记忆 | 深度知识图谱引擎 | LangGraph 原生记忆模块 |
| **架构特色** | 三段记忆分层 | 扁平向量库 | 双向时间图谱 | 知识图谱 + 向量 | 扁平 KV + 向量 + 程序记忆 |
| **上下文管理** | ✅ 全自动 | ❌ 不管 | ❌ 不管 | ❌ 不管 | ⚠️ 依赖 LangGraph |
| **自部署** | ✅ 完全自托管 | ❌ 云服务 | ⚠️ 引擎开源但需 Neo4j | ✅ 原生本地部署 | ✅ 自建 |
| **适用场景** | 对数据安全和长期记忆有刚性需求 | 快速原型验证、记忆量可控 | 时序推理、合规要求 | 多跳推理、企业知识管理 | 已用 LangGraph 的团队 |
| **数据隐私** | 最好（数据完全自控） | 较差（数据在云端） | 中等（托管版在云端） | 好（支持离线部署） | 好（数据自控） |

### 选型思路

1. 要快速上线验证 → **Mem0**
2. 要全套记忆管理 + 数据安全 → **Letta**
3. 要时序回溯能力 → **Zep**
4. 要深度知识图谱 → **Cognee** + 自己搭会话管理层
5. 已用 LangGraph → **LangMem**，零额外成本
